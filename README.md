# GitHub Analyzer

LLM-агент, который умеет отвечать на произвольные вопросы о любом публичном GitHub-репозитории, не загружая весь его код в контекст модели.

## Смысл проекта

Когда разработчик заходит в незнакомый репозиторий, он не читает его целиком. Он смотрит README, пробегает структуру папок, открывает 2–3 ключевых файла и складывает ответ в голове. GitHub Analyzer повторяет именно этот алгоритм, но руками LLM.

Главная идея — **selective context loading**: модель не получает репозиторий одним куском (и даже физически не может, репозитории не влезают в контекст и стоят денег). Вместо этого ей даётся набор инструментов, и она сама решает, что прочитать под конкретный вопрос:

- спросили про стек → агент сходит за `go.mod` / `package.json` / `Dockerfile`;
- спросили про аутентификацию → поищет файлы с `auth`, `jwt`, `middleware` и прочитает их;
- спросили про архитектуру → возьмёт дерево репо и точки входа.

Это мини-демонстрация того, как вообще устроены агентные RAG-системы над кодом: **инструменты + цикл + стратегия выбора**, а не «запихнём всё в промпт».

Заодно проект показывает:
- как собрать свой **MCP-сервер** (Model Context Protocol) и подключить его к агенту;
- как работает agentic loop на tool calling через OpenAI-совместимый API;
- как кэшировать внешние API (GitHub) в Redis, чтобы повторные вопросы по одному репо отвечались мгновенно и не жгли rate limit.

## Как это работает

```
┌──────────┐    вопрос     ┌─────────┐   tool calls   ┌─────────────┐   HTTPS   ┌──────────┐
│   CLI    │ ─────────────▶│  Agent  │ ──────────────▶│ MCP server  │ ────────▶ │  GitHub  │
│ (click)  │               │ (Yandex │                │  (stdio)    │           │   API    │
│          │◀───── ответ ──│  GPT)   │◀── tool result─│             │◀──────────│          │
└──────────┘               └─────────┘                └─────┬───────┘           └──────────┘
                                                            │
                                                            ▼
                                                      ┌──────────┐
                                                      │  Redis   │
                                                      │  (кэш)   │
                                                      └──────────┘
```

Три слоя:

1. **MCP-сервер** (`mcp_server/`) — отдельный процесс по stdio, предоставляет 4 инструмента:
   - `get_readme` — README репозитория;
   - `get_repo_structure` — дерево файлов через `git/trees/HEAD?recursive=1`;
   - `search_files` — поиск путей по подстроке;
   - `read_file` — содержимое конкретного файла (с лимитом 100 KB, чтобы случайно не влить гигабайт в контекст).

   Каждый вызов обёрнут декоратором `@cached` — результат GitHub API лежит в Redis на `CACHE_TTL` секунд.

2. **Агент** (`agent/agent.py`) — оркестратор. Поднимает MCP-сессию, забирает у сервера список тулов и прокидывает их в YandexGPT через OpenAI-совместимый эндпоинт Yandex Cloud. Крутит цикл: пока модель отвечает `tool_calls` — выполняет вызовы через MCP и возвращает результат обратно; как только модель вернула обычный текст — это финальный ответ.

   Стратегия «с чего начинать и что читать под какой вопрос» задана в `agent/prompts.py`.

3. **Интерфейс** — CLI на `click` (`cli/main.py`). MCP-сервер не привязан к CLI: его может дёргать любой клиент (Claude Desktop, другой агент, бот).

## Запуск

### Через Docker Compose

```bash
cp .env.example .env
# заполнить YANDEX_API_KEY и YANDEX_FOLDER_ID
docker compose up -d redis
docker compose run --rm analyzer python -m cli.main https://github.com/owner/repo -q "What is this project about?"
```

### Локально

```bash
pip install -e .
cp .env.example .env
redis-server &
python -m cli.main https://github.com/owner/repo -q "Какой здесь стек?"
```

### Переменные окружения

| Переменная | Обязательная | Зачем |
|---|---|---|
| `YANDEX_API_KEY` | да | ключ Yandex Cloud Foundation Models |
| `YANDEX_FOLDER_ID` | да | идентификатор каталога Yandex Cloud |
| `YANDEX_MODEL` | нет | `yandexgpt-lite` (по умолчанию) или `yandexgpt` |
| `REDIS_URL` | нет | `redis://localhost:6379` по умолчанию |
| `GITHUB_TOKEN` | нет | без токена — 60 req/hour, с токеном — 5000 |
| `CACHE_TTL` | нет | TTL кэша в секундах, по умолчанию 3600 |

## Eval

В `eval/` лежит маленький прогон по тест-кейсам: `(repo_url, question, expected_keywords)`. Проверяет, что в финальном ответе агента встречаются ожидаемые ключевые слова, и сохраняет результаты в JSON с таймстемпом.

```bash
python -m eval.run_eval
```

Смысл — не точные метрики, а регресс-проверка: после правки промпта или добавления нового тула сразу видно, не сломалось ли поведение на известных репозиториях.

## Структура

```
github-analyzer/
├── mcp_server/          # MCP-сервер с инструментами над GitHub API
│   ├── server.py        # регистрация тулов, stdio-транспорт
│   └── tools/
│       ├── github.py    # вызовы GitHub API
│       └── cache.py     # Redis-кэш (@cached декоратор)
├── agent/               # агент-оркестратор
│   ├── agent.py         # agentic loop поверх YandexGPT
│   └── prompts.py       # системный промпт со стратегией
├── cli/                 # CLI-интерфейс на click
├── eval/                # тест-кейсы и раннер
└── docker-compose.yml   # redis + analyzer
```
